# 01. 사용자 여정 맵 (User Journey Map)

> **문서 목적**: 핵심 사용 시나리오별로 사용자의 단계, 터치포인트, 감정, 개선 기회를 정리하여 UX 설계의 기반을 제공한다.
> **대상 독자**: UX 디자이너, 기획자, 개발자
> **참조**: 화면 번호는 `04-information-architecture.md`, 용어는 `06-glossary.md` 기준

## 시나리오 1: 신규 가입 → 첫 Patrol(순찰) 완료

> **페르소나**: 반려견을 키우는 30대 직장인. 동네 산책을 하면서 지역 안전에 기여하고 싶어 앱을 다운로드함.

| 단계 | 행동 | 터치포인트 (화면) | 감정 | 개선 기회 |
|------|------|------------------|------|-----------|
| 1. 앱 실행 | 앱스토어에서 설치 후 최초 실행 | U00 스플래시 | 중립 -- 기대감 | 로딩 시간 최소화 (2초 이내 자동 전환) |
| 2. 앱 소개 확인 | 온보딩 슬라이드 3장을 스와이프하며 핵심 기능 파악 | U00a 온보딩 슬라이드 | 긍정 -- "이런 앱이구나" | 슬라이드를 건너뛸 수 있는 "건너뛰기" 버튼 제공 |
| 3. 로그인 화면 진입 | 가입/로그인 방법 선택 | U00b 로그인 | 중립 | 소셜 로그인(카카오/네이버/Google/Apple)을 상단에 배치하여 가입 허들 최소화 |
| 4. 회원가입 | 이메일, 비밀번호, 닉네임, 휴대폰 번호 입력 | U00c 회원가입 | 부정 -- "입력할 게 많네" | 필수 항목 최소화, 실시간 유효성 검사로 에러 반복 방지 |
| 5. 가입 완료 → 홈 진입 | 가입 성공 후 홈(순찰 대기) 화면 자동 이동 | U02 순찰 대기 | 긍정 -- "바로 시작할 수 있구나" | 가입 축하 토스트 메시지 + 첫 Patrol 유도 CTA |
| 6. 위치 권한 요청 | GPS 권한 허용 팝업 표시 | U21 위치 권한 요청 | 중립 -- "왜 필요하지?" | 권한 요청 전 사유 설명 화면(Pre-permission) 제공: "정확한 순찰 경로 기록을 위해 위치 정보가 필요합니다" |
| 7. Patrol 시작 | '순찰 시작' 버튼 탭 → GPS 트래킹 활성화 | U02 → U03 순찰 진행 중 | 긍정 -- "시작했다!" | 시작 시 카운트다운(3-2-1) 또는 즉시 시작 옵션 제공 |
| 8. Patrol 진행 | 지도에 경로가 그려지는 것을 확인하며 산책 | U03 순찰 진행 중 | 긍정 -- "경로가 기록되고 있어" | 실시간 거리/시간/속도 표시를 크고 읽기 쉽게 배치 |
| 9. PatrolMark 생성 | 순찰 중 특이 사항을 발견하여 마킹 | U05 현장 마킹 | 긍정 -- "기록을 남길 수 있어" | 마킹 버튼을 항상 접근 가능한 위치(FAB)에 배치 |
| 10. Patrol 종료 | '종료' 버튼 탭 → 결과 화면 이동 | U03 → U07 순찰 결과 | 긍정 -- "얼마나 걸었지?" | 종료 전 확인 팝업으로 실수 종료 방지 |
| 11. 결과 확인 및 활동 일지 | 거리/시간/Points 확인, 활동 일지(Patrol.note) 입력 | U07 순찰 결과 | 긍정 -- "보람차다" | Points 획득 애니메이션으로 성취감 극대화, 일지는 선택 사항임을 명확히 표시 |

**핵심 Pain Point**: 회원가입 단계의 입력 항목이 많아 이탈 가능성 존재
**핵심 Delight**: 첫 Patrol 완료 후 Points 획득 → 즉각적인 보상 피드백

## 시나리오 2: Patrol 중 이상 발견 → Report(신고)

> **페르소나**: 정기적으로 순찰하는 40대 방범대원. 순찰 중 불법 투기 현장을 발견함.

| 단계 | 행동 | 터치포인트 (화면) | 감정 | 개선 기회 |
|------|------|------------------|------|-----------|
| 1. Patrol 진행 중 | 경로를 따라 이동하며 GPS 트래킹 중 | U03 순찰 진행 중 | 중립 -- 일상적 활동 | 현재 위치 중심 자동 추적 ON/OFF 토글 |
| 2. 이상 상황 발견 | 불법 투기 현장을 발견하고 기록하기로 결정 | U03 → U05 현장 마킹 | 부정 -- "이런 걸 발견하다니" | 마킹 버튼 접근성 확보 (순찰 화면 하단 FAB) |
| 3. PatrolMark 유형 선택 | '신고' 유형 선택 | U05 현장 마킹 (유형 선택) | 중립 | '일반 기록'과 '신고'를 시각적으로 구분 (아이콘 + 색상 차별화) |
| 4. Report 작성 | 카테고리 선택(불법투기) → 상세 설명 입력 → 사진 촬영/첨부 | U06 신고 작성 | 중립 -- "정확히 기록해야지" | 카테고리를 아이콘+텍스트로 직관적 표현, 사진 촬영 버튼 강조 |
| 5. 카메라 권한 요청 | 최초 사진 촬영 시 카메라 권한 팝업 | U22 카메라/갤러리 권한 | 중립 | 권한 거부 시 갤러리 선택으로 대체 안내 |
| 6. 사진 촬영 | 현장 사진 촬영 (최대 3장) | 시스템 카메라 → U06 | 중립 | 촬영 후 바로 미리보기 제공, 재촬영 용이하게 |
| 7. Report 제출 | '제출' 버튼 탭 → Report.status = submitted | U06 → U03 (순찰 계속) | 긍정 -- "신고 완료!" | 제출 완료 토스트 메시지 + 지도에 신고 핀(빨간색) 표시 |
| 8. Patrol 계속/종료 | 순찰을 이어서 진행하거나 종료 | U03 순찰 진행 중 | 중립 | 신고 후 자동으로 순찰 화면 복귀, 타이머 연속 |
| 9. Patrol 결과 확인 | 종료 후 마킹 수에 신고 건이 포함된 결과 확인 | U07 순찰 결과 | 긍정 -- "제대로 기록됐구나" | Report 건수를 별도로 강조 표시 |

**핵심 Pain Point**: 신고 작성 중 Patrol이 중단되지 않을까 하는 불안감
**핵심 Delight**: 신고가 즉시 접수되고 관리자에게 전달된다는 피드백

**엣지 케이스 처리**:
- **네트워크 오류 시**: Report 데이터를 로컬에 임시 저장 → 네트워크 복구 시 자동 전송 → "전송 대기 중" 상태 표시
- **사진 용량 초과 시**: 자동 압축 처리 후 업로드

## 시나리오 3: 커뮤니티 활동

> **페르소나**: 순찰 활동을 인증하고 다른 대원(User)들과 소통하고 싶은 20대 반려견주.

| 단계 | 행동 | 터치포인트 (화면) | 감정 | 개선 기회 |
|------|------|------------------|------|-----------|
| 1. 커뮤니티 탭 진입 | 하단 탭바에서 '커뮤니티' 탭 선택 | U10 피드 | 중립 | 미읽음 게시글/댓글 뱃지로 신규 콘텐츠 인지 유도 |
| 2. 카테고리 탐색 | 활동인증 / 정보공유 / 자유 탭 전환 | U10 피드 (카테고리 탭) | 중립 | 기본 탭을 '전체'로 설정하여 콘텐츠 풍부하게 노출 |
| 3. Post 목록 스크롤 | 게시글 목록을 스크롤하며 탐색 | U10 피드 | 긍정 -- "다른 대원들도 열심히 하네" | 썸네일 이미지 표시, 좋아요/댓글 수 미리보기 |
| 4. Post 상세 조회 | 관심 있는 게시글 탭 → 상세 화면 | U11 게시글 상세 | 긍정 -- "재미있다" | 이미지 스와이프 뷰어, 확대 가능 |
| 5. 좋아요 | 게시글에 좋아요 토글 | U11 게시글 상세 | 긍정 | 좋아요 시 즉각 피드백 (하트 애니메이션 + 카운트 증가) |
| 6. 댓글 작성 | 댓글 입력 후 등록 | U11 게시글 상세 (댓글 영역) | 긍정 -- "소통하는 느낌" | 댓글 등록 후 자동 스크롤, 키보드 위에 입력창 고정 |
| 7. 게시글 작성 진입 | 피드 화면에서 '글쓰기' 버튼(FAB) 탭 | U10 → U12 게시글 작성 | 중립 | FAB 버튼을 눈에 잘 띄는 위치에 배치 |
| 8. Post 작성 | 게시판(board_type) 선택 → 제목/본문 입력 → 이미지 첨부(최대 5장) | U12 게시글 작성 | 중립 -- "어떻게 쓰지" | 활동인증 카테고리 선택 시 최근 Patrol 결과 자동 첨부 옵션 |
| 9. Post 등록 | '등록' 버튼 탭 → 피드로 복귀 | U12 → U10 피드 | 긍정 -- "공유 완료!" | 등록 완료 토스트 + 피드 최상단에 내 게시글 노출 |

**핵심 Pain Point**: 게시글 작성 시 카테고리 선택이 직관적이지 않을 수 있음
**핵심 Delight**: 다른 User(대원)들의 좋아요/댓글을 받으며 소속감 형성

**엣지 케이스 처리**:
- **게시글 작성 중 앱 이탈**: 임시 저장 → 복귀 시 이어서 작성 팝업
- **이미지 업로드 실패**: 재시도 버튼 제공, 실패 이미지 표시

## 시나리오 4: 관리자 Report(신고) 처리

> **페르소나**: 지역 순찰대를 운영하는 관리자(User, role=admin). 접수된 신고를 확인하고 처리해야 함.

| 단계 | 행동 | 터치포인트 (화면) | 감정 | 개선 기회 |
|------|------|------------------|------|-----------|
| 1. 관리자 웹 로그인 | 이메일/비밀번호로 관리자 웹 로그인 | A09 로그인 | 중립 | 로그인 상태 유지(Remember Me) 옵션 |
| 2. 대시보드 확인 | 핵심 지표 확인 — 미처리 Report 건수에 주목 | A01 대시보드 | 중립 -- "오늘 신고가 몇 건이지?" | 미처리 Report 건수를 빨간색 뱃지로 강조, 클릭 시 A03으로 이동 |
| 3. 신고 관리 진입 | 사이드바에서 '신고 관리' 메뉴 클릭 | A03 신고 관리 (목록) | 중립 | 상태별 필터 기본값을 'submitted'(접수됨)으로 설정 |
| 4. Report 목록 필터링 | 상태별(submitted/in_progress/resolved) 필터 적용 | A03 신고 관리 (목록) | 중립 | 상태별 건수를 탭 위에 숫자로 표시 |
| 5. Report 상세 조회 | 특정 신고 건 클릭 → 상세 화면 | A03 신고 관리 (상세) | 중립 -- "어떤 상황이지?" | 사진 확대 뷰 + 지도에 위치 핀 표시 + 신고자 정보 표시 |
| 6. 상태 변경: 처리 중 | Report.status를 submitted → in_progress로 변경 | A03 신고 관리 (상세) | 중립 | 상태 변경 시 확인 팝업, 변경 사유(admin_note) 입력 옵션 |
| 7. 조치 진행 | 실제 현장 조치 (앱 외부 활동) | - | 중립 | - |
| 8. 상태 변경: 처리 완료 | Report.status를 in_progress → resolved로 변경 + 처리 메모 입력 | A03 신고 관리 (상세) | 긍정 -- "처리 완료" | 처리 완료 시 신고자에게 자동 PushNotification 발송 옵션 |
| 9. CSV 추출 | 월별 신고 데이터를 CSV로 다운로드 | A03 신고 관리 (목록) | 중립 | 기간 필터 + 원클릭 CSV 다운로드 |

**핵심 Pain Point**: 다수의 Report를 하나씩 처리하는 반복 작업의 피로감
**핵심 Delight**: 신고 처리 완료 시 신고자에게 자동 알림 → 관리 효율성

**개선 기회**:
- 일괄 상태 변경 기능 (체크박스 선택 → 일괄 처리)
- Report 건수 추이 차트로 지역 안전 현황 파악

## 시나리오 5: PushNotification(알림) 수신 → 확인 → 이동

> **페르소나**: 평소 앱을 사용하는 30대 User(대원). 관리자가 발송한 이벤트 알림을 수신함.

| 단계 | 행동 | 터치포인트 (화면) | 감정 | 개선 기회 |
|------|------|------------------|------|-----------|
| 1. Push 수신 | 기기 잠금 화면 또는 상단 알림 배너에서 PushNotification 수신 | OS Push Notification (시스템 레벨) | 중립 -- "뭐지?" | 알림 제목을 명확하게 작성 (예: "[이벤트] 주말 보너스 2배 순찰!") |
| 2-A. 알림 탭 (앱 미실행 시) | Push 알림 탭 → 앱 실행 → 딥링크 대상 화면으로 이동 | OS → U00 스플래시 → 딥링크 대상 화면 | 중립 | 스플래시 최소화 후 즉시 대상 화면 전환, 로그인 만료 시 로그인 후 대상 화면으로 리다이렉트 |
| 2-B. 알림 탭 (앱 실행 중) | Push 배너 탭 → 딥링크 대상 화면으로 직접 이동 | 현재 화면 → 딥링크 대상 화면 | 중립 | 현재 작업 중단 경고 없이 부드러운 전환, 뒤로가기 시 원래 화면 복귀 |
| 3. 알림 목록 확인 | 헤더 벨 아이콘 탭 → 알림 목록 진입 | U09 알림 목록 | 중립 | 미읽음 알림 강조 표시 (굵은 글씨 + 파란 점), 읽음 처리 시 시각적 변화 |
| 4. 알림 상세 확인 | 특정 알림 항목 탭 → 관련 화면으로 이동 | U09 → 딥링크 대상 화면 | 긍정 -- "이벤트가 있구나!" | 알림 유형별 아이콘 차별화 (이벤트/신고처리결과/공지 등) |
| 5. 대상 화면에서 행동 | 이벤트 상세 확인 또는 순찰 시작 등 후속 행동 | 딥링크 대상 화면 (예: U02 순찰 대기) | 긍정 -- "참여해볼까" | 이벤트 기간 동안 홈 화면에 이벤트 배너 상시 노출 |

**핵심 Pain Point**: 알림이 너무 많으면 무시하게 됨 → 발송 빈도 관리 필요
**핵심 Delight**: 딥링크를 통한 원클릭 이동 → 관심 콘텐츠에 즉시 접근

**엣지 케이스 처리**:
- **Push 권한 거부 상태**: 설정 화면(U20)에서 알림 ON/OFF 안내, OS 설정 이동 딥링크 제공
- **딥링크 대상 화면이 삭제된 경우**: "해당 콘텐츠가 삭제되었습니다" 안내 후 홈으로 이동
- **로그인 만료 상태에서 딥링크 진입**: 로그인 화면(U00b) → 로그인 성공 → 원래 딥링크 대상 화면으로 리다이렉트

## 여정 맵 종합 인사이트

### 공통 Pain Points
1. **권한 요청 피로**: GPS, 카메라, Push 등 다수의 권한 요청이 초기에 몰림 → 필요 시점에 순차적으로 요청
2. **네트워크 의존성**: Patrol 중 네트워크 불안정 시 데이터 유실 불안 → 오프라인 우선 설계(로컬 저장 → 동기화)
3. **긴 Patrol 세션**: 장시간 Patrol 시 배터리 소모 걱정 → 배터리 최적화 + 저전력 모드 안내

### 공통 Delight Moments
1. **Points 획득**: Patrol 종료 시 즉각적인 Points 보상 피드백
2. **Ranking 상승**: 누적 활동으로 Ranking(리더보드) 순위가 올라가는 성취감
3. **커뮤니티 반응**: Post에 좋아요/댓글이 달리는 사회적 보상
4. **Report 처리 알림**: 내가 신고한 건이 처리되었다는 PushNotification → 기여 실감

### UX 설계 원칙 제안
1. **오프라인 우선**: Patrol 데이터는 항상 로컬에 먼저 저장, 네트워크 복구 시 동기화
2. **점진적 권한 요청**: 기능 사용 시점에 해당 권한만 요청 (선제적 일괄 요청 지양)
3. **즉각적 피드백**: 모든 사용자 액션에 시각적/촉각적 피드백 제공
4. **최소 입력**: 자동 입력 가능한 항목(GPS 좌표, 시간 등)은 시스템이 자동 처리
5. **에러 복구**: 모든 에러 상황에서 사용자가 다음 단계를 알 수 있는 안내 제공
