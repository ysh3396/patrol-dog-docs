반려견순찰대 앱 개발 제안서

발표 대본 및 예상 Q&A

슬라이드 21개 예상 질문 11개 발표 약 20–25분

발표 대본

슬라이드별 낭독 스크립트 — 녹색 인용 블록이 실제 발화 내용입니다

1

타이틀

안녕하세요. 반려견순찰대 GPS 트래킹 및 커뮤니티 앱 프로젝트 제안서 발표를 시작하겠습니다.

이 프로젝트는 모바일 앱(iOS/Android)과 관리자 웹으로 구성된 서비스로, 총 37개 화면, 4대 핵심 기능 영역, 그리고 실시간 GPS 트래킹을 중심으로 설계되었습니다.

2

프로젝트 배경

현재 도시의 안전 사각지대는 점점 늘어나고 있지만, 기존 방범 활동은 종이 기록과 수기 출석 체크에 머물러 데이터 활용이 불가능한 상태입니다.

그런데 반려견주는 매일 1~2회 산책을 합니다. 이 산책 경로가 곧 지역 방범 순찰 경로가 될 수 있다는 점에 착안했습니다.

순찰 거리, 시간, 횟수를 자동으로 집계하는 도구가 없어 활동량 파악이나 우수 대원 평가가 어려운 현실입니다.

반려견순찰대 앱은 개인의 건강관리, 동네 방범, 커뮤니티 소속감을 하나의 앱으로 통합하는 사회적 가치를 가진 플랫폼입니다.

3

프로젝트 목적

이 프로젝트의 핵심 목적은 4가지입니다.

첫째, 실시간 GPS 트래킹으로 산책 및 순찰 경로를 기록하고 지도 위에 시각화합니다.
둘째, 순찰 거리·시간·횟수를 자동 집계하여 개인, 팀, 지역 단위 통계를 제공합니다.
셋째, 활동 인증, 정보 공유, 자유 게시판으로 대원 간 소통과 동기부여를 촉진합니다.
넷째, 포인트와 랭킹 시스템으로 지속적인 참여를 유도합니다.

목표 KPI로는 3개월 내 MAU 1,000명, 주 3회 이상 평균 순찰 횟수, 회당 평균 2km 이상 순찰 거리, 그리고 7일 이탈률 30% 이하를 설정했습니다.

4

4대 기능 영역 개요

핵심 기능은 크게 4가지 영역으로 나뉩니다.

순찰 트래킹 시스템 — GPS 실시간 경로 기록, 백그라운드 실행, 현장 마킹과 신고 기능입니다. P0 필수 기능입니다.

포인트 & 게이미피케이션 — 1km당 100포인트 자동 지급, 실시간/주간/월간 리더보드, 어뷰징 방지 시스템을 포함합니다.

커뮤니티 & 미디어 — 활동인증, 정보공유, 자유 게시판과 Push 알림, 영상 콘텐츠 기능입니다.

관리자 웹 대시보드 — 대원 관리, 신고 처리, 이벤트 등록, 통계 CSV 추출 등 운영 기능을 제공합니다.

5

순찰 트래킹 시스템 상세

순찰 트래킹의 핵심 기능을 자세히 보겠습니다.

GPS 실시간 트래킹은 P0 필수 기능으로, 지도 위에 폴리라인으로 경로를 실시간 렌더링하고, 거리 누적, 경과 시간, 이동 속도를 표시합니다.

백그라운드 실행 역시 P0입니다. 5~10미터 이동 시 또는 10초 간격으로 위치를 수집하며, iOS는 Background Location Updates, Android는 Foreground Service를 활용합니다. GPS 신호 유실 시 자동 복구되고, 로컬 버퍼링 후 배치 전송으로 배터리를 최적화합니다.

현장 마킹은 사진 최대 3장과 메모 200자를 기록할 수 있고, GPS 좌표가 자동으로 기록됩니다.

활동 일지에서는 순찰 결과를 거리, 시간, 마킹 수, 포인트로 요약하고, 30분 이상 정지 시 자동 종료를 안내합니다.

6

포인트 & 게이미피케이션

포인트 시스템은 1km = 100포인트로 단순명확합니다. 0.1km 단위로 비례 계산하며, 일일 최대 1,000포인트 상한이 있습니다. 이벤트 시 보너스 배율도 적용 가능합니다.

랭킹은 실시간, 주간, 월간 3종으로 운영하며, 누적 순찰 거리 기준입니다.

어뷰징 방지가 중요한데요. 도보 6km/h 초과 시 경고, 자전거 25km/h 초과 시 경고, 차량 40km/h 이상이면 포인트 미지급으로 처리합니다. 위치 점프(순간이동)도 감지하여 해당 구간을 무효 처리합니다. 정확한 임계값은 베타 테스트 후 확정할 예정입니다.

7

커뮤니티 & 미디어

커뮤니티는 활동인증, 정보공유, 자유 게시판 3개로 구성됩니다. 이미지 최대 5장 첨부, 댓글과 좋아요, 부적절 게시글 신고 기능을 갖추고 있습니다.

동영상 콘텐츠는 P2 부가 기능으로, YouTube나 Vimeo를 임베드하는 방식이라 자체 호스팅 비용이 없습니다. 관리자만 영상을 등록할 수 있습니다.

Push 알림은 FCM 단일 연동으로 iOS와 Android를 동시에 지원합니다. 관리자 웹에서 전체 또는 특정 그룹 대상으로 발송하며, 딥링크로 해당 화면으로 바로 이동합니다.

8

관리자 웹

관리자 웹은 6개 핵심 메뉴로 구성됩니다.

대시보드에서 총 대원 수, 금일/금주 순찰 횟수, 누적 거리, 미처리 신고 등 KPI를 한눈에 볼 수 있습니다.
대원 관리에서 검색, 필터, 정렬과 개별 순찰 이력 조회, CSV 다운로드가 가능합니다.
신고 관리에서는 접수 → 처리중 → 해결 상태 전이를 관리합니다.
그 외 이벤트 관리, 포인트 수동 지급/차감, 기간별 통계와 히트맵 기능이 있습니다.

9

정보 구조 (IA)

앱의 정보 구조를 보겠습니다.

모바일 앱은 4개 탭 구조입니다. 홈(순찰) 탭에 7개 화면, 커뮤니티 탭에 4개, 랭킹 탭 1개, 마이페이지 탭에 6개 화면이 배치됩니다. 여기에 온보딩과 공통 화면을 합쳐 총 27개입니다.

관리자 웹은 대시보드부터 설정까지 10개 화면으로 구성됩니다.

10

전체 화면 목록

이 슬라이드에서 전체 37개 화면 목록을 확인하실 수 있습니다.

앱 27개 화면은 U00부터 U23까지, 관리자 웹 10개는 A01부터 A10까지 번호가 부여되어 있어 향후 개발 시 화면 단위로 명확하게 커뮤니케이션이 가능합니다.

11

와이어프레임 프로토타입

와이어프레임 프로토타입은 브라우저에서 바로 확인 가능한 인터랙티브 형태로 제작했습니다.

앱 프로토타입과 관리자 프로토타입을 각각 분리하여 제공하며, 개발사에서 실제 구현 시 참고하실 수 있도록 ASCII 와이어프레임 형식으로 설계했습니다.

이 시점에서 실제 프로토타입을 브라우저에서 시연하면 설득력이 크게 올라갑니다
12

앱 프레임워크 비교

기술 스택 선정을 위해 3가지 프레임워크를 비교했습니다.

React Native, Flutter, Native(Kotlin+Swift) 세 가지입니다.

GPS 백그라운드 안정성은 세 옵션 모두 충분하지만, 개발 비용과 기간 면에서 크로스플랫폼이 유리합니다. React Native는 OTA 업데이트(CodePush)가 가능하다는 큰 장점이 있어, 버그 발생 시 앱 스토어 심사 없이 바로 배포할 수 있습니다.

React Native를 추천합니다. JS/TS 팀으로 빠른 MVP 출시가 가능하고, GPS 백그라운드 트래킹은 검증된 서드파티 모듈이 존재합니다.

13

서버 & DB 비교

서버 프레임워크는 Node.js, Spring Boot, NestJS를 비교했습니다. Node.js가 실시간 처리, 개발 속도, JS 풀스택 구성 면에서 가장 적합합니다.

데이터베이스는 PostgreSQL + PostGIS를 추천합니다. GPS 경로 데이터의 공간 쿼리 성능이 최강이고, 포인트 지급 같은 금전성 트랜잭션에 필요한 ACID를 완벽히 지원합니다.

14

인프라 & 외부 API

지도 API는 네이버 지도를 추천합니다. 국내 서비스 특성상 한국 도로와 건물 데이터 정확도가 최우선이기 때문입니다.

클라우드 인프라는 AWS 서울 리전입니다. 풍부한 관리형 서비스(RDS, ECS, Lambda)로 운영 부담을 줄이고, 오토스케일링으로 사용자 증가에 유연하게 대응할 수 있습니다.

15

시스템 아키텍처

전체 시스템 아키텍처입니다.

클라이언트 레이어에 iOS앱, Android앱, 관리자 웹이 있고, HTTPS와 WebSocket으로 API Gateway에 연결됩니다. API Gateway에서 인증 검증, 라우팅, Rate Limiting을 처리합니다.

서비스 레이어에는 App Server(위치수신, 순찰, 커뮤니티, 포인트, 인증), Push Service(FCM/APNs), Map API(네이버 지도)가 있습니다.

데이터 레이어는 PostgreSQL + PostGIS로 사용자, 순찰 경로, 위치 이력, 커뮤니티, 포인트 데이터를 관리합니다.

16

GPS 백그라운드 전략

GPS 백그라운드 트래킹은 이 앱의 가장 핵심이자 기술적 난이도가 높은 부분입니다.

iOS에서는 pausesLocationUpdatesAutomatically = false 설정이 핵심입니다. 이걸 true로 두면 iOS가 임의로 수집을 중단하는데, 이것이 가장 흔한 이슈입니다. OS 강제 종료 시에는 significantLocationChange로 앱을 재시작하여 트래킹을 자동 복원합니다. App Store 심사를 위해 파란 바 표시와 명확한 사유 설명이 필요합니다.

Android에서는 Foreground Service로 상단 알림을 상시 표시하여 OS가 앱을 강제 종료하지 않도록 합니다. Android 10, 12, 14 각 버전별 권한 대응도 설계에 포함했습니다.

공통 전략으로 5m 이동 또는 10초 간격 수집, 로컬 버퍼링 후 30초~1분 배치 전송, 1시간 순찰 기준 약 8~12% 배터리 소모를 예상합니다.

17

iOS Live Activity & Dynamic Island

차별화 기능으로 iOS Live Activity와 Dynamic Island를 제안합니다.

잠금 화면에서 앱을 열지 않아도 순찰 거리, 시간, 속도를 바로 확인할 수 있고, Dynamic Island에서 일시정지와 종료도 가능합니다.

이 기능은 App Store 심사에서도 유리합니다. 백그라운드 위치 사용의 명확한 가치를 증명할 수 있기 때문입니다.

iOS 16.1 이상에서만 지원되며, 이전 버전은 기존 알림으로 폴백합니다.

18

개발 범위 요약

개발 범위를 정리하면, P0 필수 기능은 GPS 트래킹, 백그라운드 실행, 포인트 시스템, 대원 관리입니다. 이것이 MVP 핵심입니다.

P1 중요 기능은 마킹, 신고, 랭킹, 커뮤니티, Push 알림 등 런칭 목표 기능이고, P2 부가 기능인 동영상은 런칭 후 추가합니다.

전체 화면 수는 앱 27개 + 관리자 웹 10개 = 총 37개입니다.

19

제안 기술 스택 (최종)

최종 추천 기술 스택을 정리하겠습니다.

앱: React Native — iOS/Android 동시 개발, 코드 공유율 70~80%, OTA 업데이트 가능
서버: Node.js — JS 풀스택, 비동기 I/O로 다수 GPS 동시 처리
DB: PostgreSQL + PostGIS — 공간 쿼리와 ACID 트랜잭션
인프라: AWS 서울 리전 — ECS/RDS 관리형 서비스
지도: 네이버 지도 — 국내 최고 데이터 정확도
Push: FCM — 단일 연동으로 양 플랫폼 지원

첫째, 수집 간격을 최적화했습니다. 5미터 이동 또는 10초 간격 중 먼저 충족되는 조건으로 수집하여 불필요한 수집을 줄입니다.

둘째, 로컬 버퍼링 후 배치 전송 방식을 사용합니다. 매번 서버로 전송하지 않고 로컬에 저장한 뒤 30초~1분 단위로 묶어서 전송하므로 네트워크 사용량이 크게 줄어듭니다.

1시간 순찰 기준 약 8~12% 배터리 소모를 예상하고 있으며, 이는 네이버 지도나 카카오내비 같은 위치 기반 앱과 유사한 수준입니다. 베타 테스트에서 실측 후 추가 최적화할 예정입니다.

충분히 우려하실 수 있는 부분입니다. 저희는 세 가지 전략을 마련했습니다.

첫째, showsBackgroundLocationIndicator = true 설정으로 상단에 파란 바를 표시합니다. Apple이 권장하는 투명성 가이드라인을 준수하는 것입니다.

둘째, Live Activity를 구현합니다. 잠금 화면에서 순찰 상태를 실시간으로 보여주므로, 백그라운드 위치 사용의 명확한 가치를 증명할 수 있습니다.

셋째, 심사 메모에 "방범 순찰 목적의 GPS 경로 기록 앱"이라는 명확한 사유를 기재합니다. 건강/피트니스 카테고리의 러닝 앱들과 동일한 사용 패턴이므로 심사 통과에 문제가 없을 것으로 판단합니다.

완벽한 방지는 어렵지만, 다층 방어 전략을 적용합니다.

  • 1차 방어: 속도 임계값 — 도보 6km/h, 자전거 25km/h 초과 시 경고, 차량 40km/h 이상 시 포인트 미지급
  • 2차 방어: 위치 점프 감지 — 순간이동(GPS 조작) 시 해당 구간 무효 처리
  • 3차 방어: 일일 최대 1,000포인트 상한으로 극단적 어뷰징의 실익 제한

정확한 임계값은 베타 테스트에서 실제 데이터를 수집한 후 확정합니다. 또한 관리자가 의심 활동을 조회하고 포인트를 수동 차감할 수 있는 관리 기능도 포함되어 있습니다.

Flutter도 충분히 좋은 선택입니다. 두 프레임워크 모두 GPS 백그라운드를 지원하고 크로스플랫폼 개발이 가능합니다.

React Native를 추천한 이유는 크게 세 가지입니다.

  • OTA 업데이트(CodePush)가 가능합니다. GPS 트래킹 앱 특성상 다양한 디바이스에서 예기치 않은 버그가 발생할 수 있는데, 앱 스토어 심사 없이 즉시 수정 배포할 수 있습니다.
  • npm 생태계가 훨씬 크기 때문에 서드파티 모듈 선택지가 많습니다.
  • 서버도 Node.js를 사용하면 프론트-백 JS 풀스택으로 팀 구성이 효율적입니다.

다만, 개발사의 팀 역량에 따라 Flutter가 더 적합할 수 있으므로, 기술 스택 확정 단계에서 개발사와 함께 최종 결정하는 것을 권장합니다.

국내 서비스라는 점이 핵심입니다.

네이버 지도는 한국 내 도로, 건물, 골목길 데이터의 정확도가 가장 높습니다. 순찰 경로가 주로 골목길과 주택가인 점을 고려하면, 세밀한 지도 데이터가 매우 중요합니다.

Google Maps는 글로벌 확장을 고려한다면 좋은 선택이지만, 현 단계에서는 국내 지역 방범이 목적이므로 네이버 지도가 최적입니다.

향후 글로벌 확장 시에는 지도 API를 교체하거나 병행할 수 있도록 추상화 레이어를 설계에 반영할 수 있습니다.

매우 중요한 질문입니다. 위치 데이터는 개인정보보호법상 민감 정보에 해당합니다.

  • 위치기반서비스사업 신고가 필요합니다. 방송통신위원회에 사업 신고를 진행해야 합니다.
  • 개인정보처리방침에 위치 데이터 수집 목적, 보관 기간, 제3자 제공 여부를 명시합니다.
  • 기술적 보호 조치로 데이터 전송 시 TLS 암호화, 서버 저장 시 위치 데이터 익명화/가명화 옵션을 제공합니다.
  • 사용자가 언제든 위치 기록 삭제를 요청할 수 있는 기능을 마이페이지에 포함합니다.

이 부분은 개발 착수 전 법률 자문을 받아 정책을 확정하는 것을 권장합니다.

AWS의 관리형 서비스를 활용하여 오토스케일링을 적용합니다.

GPS 데이터 수집은 로컬 버퍼링 후 배치 전송이므로 실시간 부하가 분산됩니다. 30초~1분마다 한 번 서버에 전송하는 구조라서, 동시 순찰자 1,000명 기준으로도 초당 요청 수는 관리 가능한 수준입니다.

ECS(컨테이너) 기반으로 트래픽 증가 시 자동으로 서버 인스턴스를 추가하고, RDS Read Replica로 읽기 쿼리를 분산합니다.

초기에는 소규모로 시작하고, MAU가 증가함에 따라 점진적으로 인프라를 확장하면 됩니다. 비용도 사용량 기반이라 초기 부담이 적습니다.

이 제안서는 기획/설계 단계의 산출물이며, 구체적인 개발 기간과 비용은 개발사의 팀 규모와 역량에 따라 달라집니다.

다만, 참고 기준을 말씀드리면:

  • P0 MVP (GPS 트래킹 + 포인트 + 관리자 기본): 약 3~4개월 예상
  • P1 포함 (커뮤니티, 랭킹, 마킹, Push 등): 추가 2~3개월
  • 안정화 및 QA: 1~2개월

총 6~9개월 정도를 예상하지만, 이는 풀타임 3~4인 팀 기준이며 개발사와 구체적으로 협의해야 합니다.

이 제안서에는 화면 목록 37개, 와이어프레임, 기능 명세가 모두 포함되어 있어 개발사가 정확한 견적을 산출하는 데 충분한 자료가 될 것입니다.

이 앱은 기본적으로 공익/사회 서비스 성격이 강합니다. 직접적인 수익 모델보다는 지자체 예산 또는 후원 기반으로 운영되는 것이 일반적입니다.

다만, 향후 확장 가능한 수익 모델은:

  • 지자체 용역/SaaS 모델: 각 시/구에 관리자 웹을 제공하고 월정액 또는 연단위 계약
  • 포인트 교환: 지역 상점 할인 쿠폰, 지자체 혜택과 연계하여 광고/제휴 수익
  • 프리미엄 기능: 상세 통계 분석, 데이터 리포트 등 부가 서비스
  • 데이터 활용: 익명화된 순찰 커버리지 데이터를 방범 정책 수립에 활용 (지자체 대상 B2G)

기존 러닝앱(Nike Run, 스트라바 등)은 개인 운동 기록에 초점이 맞춰져 있고, 기존 방범 앱은 단순 출석 체크 수준입니다.

반려견순찰대 앱의 핵심 차별점은:

  • 개인의 일상 산책을 지역 사회 안전 활동으로 연결하는 유일한 플랫폼입니다.
  • GPS 트래킹 + 포인트 + 커뮤니티 + 관리자 대시보드가 하나의 통합 시스템입니다.
  • 어뷰징 방지와 공정한 랭킹으로 건전한 경쟁 환경을 만듭니다.
  • 관리자 웹에서 통계·신고·이벤트를 관리할 수 있어 행정 효율성을 높입니다.

빠른 프로토타이핑에는 Firebase나 MongoDB가 유리한 것이 맞습니다.

하지만 이 앱에는 두 가지 핵심 요구사항이 있습니다.

  • 공간 쿼리: GPS 경로를 저장하고 "특정 영역 내 순찰 커버리지 분석" 같은 쿼리를 수행해야 하는데, PostGIS는 이 분야에서 업계 최고입니다.
  • 트랜잭션: 포인트 지급은 금전적 가치를 가지므로, "순찰 완료 → 거리 계산 → 포인트 지급"이 원자적(atomic)으로 처리되어야 합니다. PostgreSQL의 ACID 트랜잭션이 이를 안정적으로 보장합니다.

MongoDB 4.0 이상도 트랜잭션을 지원하지만, 공간 쿼리 성능과 트랜잭션 안정성 모두에서 PostgreSQL이 이 프로젝트에 더 적합합니다.
